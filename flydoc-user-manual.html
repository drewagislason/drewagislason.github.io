flydoc User Manual

Version 1.0, Feb 29, 2024

flydoc is a cmdline tool which creates developer documentation from commented source code.

Like markdown, flydoc is minimalist. It creates static web pages that can then be hosted anywhere, or markdown files that can be turned into .pdfs.

flydoc works with a variety of coding languages, including: C, C++, C#, Go, Java, Javascript, Python, Ruby, Rust, Swift and Typescript. Basically any language that uses # or /* or // for comments and has functions/methods that use parentheses. It can also work with other languages with a bit more source documenting.

flydoc uses @keywords in special comments to control the look and content of the documentation. The output is either a single markdown file or a folder of .html files. The markdown files can easily be turned into .pdfs using tools like Mou, MacDown or Typora. The HTML uses W3.CSS for styling.

flydoc is mobile first, that is it looks great in the browser of a smart phone, tablet or desktop.

For an example of flydoc output, see https://drewagislason.github.io/index.html
For a markdown tutorial, see https://www.markdownguide.org/basic-syntax/
For an online markdown converter, see https://markdowntohtml.com
For markdown editors, see: https://www.makeuseof.com/best-markdown-editors-windows/
For a W3.css tutorial, see https://www.w3schools.com/w3css/default.asp
For popular coding languages, see: https://madnight.github.io/githut/#/pull_requests/2021/3

flydoc is named after https://en.wikipedia.org/wiki/Firefly_(TV_series), an iconic space western.

Features

  • Create developer API documentation from commented source code documentation
  • Outputs markdown or static HTML/CSS with links
  • Mobile first HTML: looks great on a phone, tablet or desktop
  • Works with any coding language
  • Converts standard markdown files to HTML
  • Control what is documented and what is not (public vs private APIs)
  • Adjust colors, logo for project/company specific look
  • Outputs statistics (# of modules, classes, functions, examples documented)
  • Warnings are in standard file:line:col: warning: text format for easy parsing

Usage

flydoc runs in the command-line terminal. Someday, perhaps, I'll make a macOs, Linux, and Windows GUI app version. How to use a terminal is behond the scope of this document.

Usage = flydoc [-n] [-o out] [-s] [-v] [--ext .c.js] [--html] [--link prefix] in...

Options:
-n       parse inputs only, no output
-o       output file (file.md) or folder (to contain .html files)
-v       verbose
-s       sort methods, modules, classes
--ext    list of file extensions to search. Default ".c.c++.cpp.cc.cs.go.java.js.md.py.rb.rs.swift.ts"
--html   create html rather than markdown
in...    Input files and folders

Examples:

$ flydoc my_api.c
$ flydoc --html -o myfolder src/*.c docs/*.md
$ flydoc --html -o apidocs pysrc/ rustsrc/ src/
$ flydoc -v -n --html -o myfolder src/*.c
$ flydoc --html -o . myfile.md
$ flydoc -v -o myfile.md src/ lib/

Flydoc searches source files for special comments marked for flydoc.

Basics

flydoc uses an opt-in approach that looks for @keywords inside of special comments. The special comments are a slight addition to standard comments you probably already have in your source code. These comments are expected to be markdown in format.

The source code documentation is gathered into modules and classes. Each module gets its own HTML file if the --html option is used, or it's own chapter if markdown output.

The results are pages (or chapters) of functions organized by class or module. A modules and classes can span multiple source files.

A very simple example follows:

/*!
  @defgroup maths   A math library to eventually rival MatLab

  More descriptive text here...
*/

/*!
  Calculate area of a circle
  @param  r   radius
  @return area of the circle
*/
double maths_circle_area(double r)

/*!
  Calculate volume of a sphere
  @param  r   radius
  @return volume of the sphere
*/
double maths_sphere_volume(double r)

/*!
  Calculate surface area of a sphere
  @param  r   radius
  @return surface area of the sphere
*/
double maths_sphere_surface_area(double r)

See also: Documenting Source Code with flydoc

Note: If you are familiar with Doxygen, many of the keywords are the same.

Keywords

The flydoc @keywords are as follows:

Keyword Usage Description
@class name description define a class
@color sidebar {titlebar} define colors side bar and title bar
@defgroup name description define a module section
@example title a markdown code block must follow
@fn prototype document a function
@inclass name all that follows is in this class
@ingroup name all that follows is in this module
@logo link define logo img for whole project (e.g. file.png)
@mainpage title mainpage with title
@param name description function/method parameter
@return description function/method return value
@version version version string for the whole project

@class name description

The @class keyword defines a class, which has a collection of methods.

/*!
    @class FlyWin  A full-color, overlapping windowing system for ANSI terminals

    Text and examples here to describe the class in detail...
*/

@color sidebar {titlebar}

You can customize the 2 HTML colors that are used to decorate the side bar and title bar. The defaults are w3-blue and w3-black. The main page has only one color and so uses the sidebar color for its titlebar.

The titlebar color is optional and will default to w3-black.

The @color keyword can appear in an @class, @defgroup or @mainpage, allowing the color schemes to be different for those pages. The @mainpage colors will be reflected to all pages that don't define their own colors.

Color has no effect on markdown output and is removed.

/*!
  @mainpage   My Project
  @color      w3-purple w3-orange
*/

For color codes, see: https://www.w3schools.com/w3css/w3css_colors.asp

For more customization, see also @logo

@defgroup name description

The @defgroup keyword defines a module, with collection of functions.

/*!
    @defgroup FlyWin  A full-color, overlapping windowing system for ANSI terminals

    Text and examples here to describe the class in detail...
*/

@example title

The @example keyword defines an example by title. The title must be unique within the module or class.

Examples must be in a module, class or function (method) definition. Below shows 3 examples, one in a module one in a function and one stand-alone that will appear in that same module in its own section.

The @example line must be followed by a markdown code block, either with triple ticks indented by 4 spaces or a tab.

Example will be listed and linked on the main HTML page (index.html).

/*!
  @ingroup    MyModule
  @example    My module example

      Example code here...
*/

/*!
  My function description
  @example    My function example

      Example code here....

  @param      n   Number of iterations
  @returns    none
*/
void my_function(unsigned n)
{
  // code here...
}

/*!
  @example  My stand-alone example still in MyModule

      Example code here...
*/

@fn prototype

flydoc can figure out the function prototype for all C-type languages like C, C++, Python, Rust, Javascript, etc...

But, if it cannot, you can explicitely give the prototype

##! @fn       backup_database()
#   @return   {Integer} status
#
def backup_database
  # code here...
end

@inclass name

While @class on a particular name must be only defined once per project, @inclass can be used anywhere. All comment lines processed from that point are considered in that class, until an @class, @defgroup or @ingroup are used.

Perhaps class methods are spread across multiple source files. In this case, use one @class to define the class, then @inclass in the other source files to indicate they are part of that same class.

/*!
  @inclass  MyClass
  @example  My Example for MyClass

      code here...
*/

@ingroup name

While @defgroup on a particular name must be only defined once per project, @ingroup can be used anywhere. All comment lines processed from that point are considered in that module (group).

Perhaps a module's functions are spread across multiple source files. In this case, use one @defgroup to define the module, then @ingroup in the other source files to indicate they are part of that same module.

/*!
  @ingroup  MyModule
  @example  My Example for MyModule

      code here...
*/

@logo link

A logo is some image you want to see in upper left of the title bar. This makes the pages look more professional. This can be a company logo, a project logo or any image you want.

The logo link is in standard markdown image link form ![text](imagefile.jpg). However, if the link [text] contains any w3- codes, those are applied to the logo. For example, the logo below will show as a circle, even though the .png is a square.

An @logo in the @mainpage applies to all pages. An @logo in an @class or @defgroup applies only to that HTML page. This allows you to use different graphics for various modules and classes.

Logos are ignored for markdown output.

/*!
  @logo ![w3-circle w3-grayscale-max](flydoclogo.png)
*/

See: https://www.w3schools.com/w3css/w3css_images.asp

@mainpage title

Any given project must have only a single @mainpage, regardless of how many modules, classes, functions, methods and examples there are. This is the high-level naviation page and describes the project. If there is more than one @manpage in the input files, then only the 1st one encountered is used. The others are ignored.

In markdown output, this becomes the document main title and section. In HTML output, this is the index.html.

/*!
  @mainpage   FireFly C Library and Tools
  @version    0.95.3
  @logo       ![w3-circle w3-grayscale-max](flydoclogo.png)

  A set of C algorithms and tools to make developing C or C++ programs easier
*/

@param name description

Methods and functions may have parameters. Describe them here. The name is the variable name, the description is a short one-line description. This is ignored if not in a function comment.

/*!
  Returns pi to n digits in fixed point.
  @param  nDigits   Number of digits after the "3", e.g. 5 to get 3.14159 (314159)
  @return The number pi in fixed point (e.g. 314 == 3.14)
*/
unsigned long pi_fixed_point(unsigned nDigits)
{
  // code here...
}

@return description

Methods and functions may return values. Describe what is returned here. @returns is the same as @return.

/*!
  Returns pi to n digits in fixed point.

  pi is already precalulated, so this is very fast. With a 64-bit longs, max nDigits are 18
  (any thing higher will be ignored): 3141592653589793238

  @param  nDigits   Number of digits after the "3", e.g. 5 to get 3.14159 (314159)
  @return The number pi in fixed point (e.g. 314 == 3.14)
*/
unsigned long pi_fixed_point(unsigned nDigits)
{
  // code here...
}

@version version

The @version is displayed in the upper left under the logo (if any). If the @version is on the @mainpage, then it will displayed on all @class and @defgroup HTML pages. Each @class or @defgroup may have their own version string. If no version string is given in the project, then none is displayed.

The version string can be anything. An example might be 1.0.222, but could be foo bar. The example below shows 0.9.gitsha.

/*!
  @version  0.9.6df33b8
*/

Warnings and errors

flydoc will produce warnings in the standard format if something is wrong with the input comments.

$ flydoc -o mytest.md test.*
test.c:7:10: warning: unknown keyword
test.py:99:10: warning: no class or module
test.rs:100:1: warning: duplicate class or module
test.cs:222:1: warning: no function found after comment

If there are warnings, your HTML or markdown documentation may look or link poorly.

Advanced topics

While hopefully flydoc is pretty clear by now, here are some more advanced flydoc topics.

Links

All HTML links produced by flydoc are predictable. All spaces are converted to dashes. Case is preserved. Classes and examples get the prefix "class_" and "example_" respectively.

HTML has various rules for special characters with references (links). It's complicated, and depends on the version of your browser. Keep to alphanumeric, spaces and underscores as much as possible.

See https://stackoverflow.com/questions/1236856/in-the-dom-are-node-ids-case-sensititve

Examples of links:

This Becomes link
MyModule MyModule.html
MyClass class_MyClass.html
MyClassMethod class_MyClass.html#MyClassMethod
My Example Title #example_My-Example-Title
my_func #my_func
myMethod #myMethod

That is @class MyClass description produces file class_MyClass.html. If there is a method in that class called MyClassMethod, then the link to it locally would be #MyClassMethod. The link to it from another class or module would be class_MyClass.html#MyClassMethod.

If you have a heading called # My Heading 1, in the file mymarkdown.md, then you could link to it from a class or module with mymarkdown.html#My-Heading-1.

An example of adding links to your function documentation:

/*!
  Copy src string to the destination.
  @param    dst   destination string at least as large as src
  @param    src   source asciiz string
  @return   ptr to dst string

  See also: [strcat](#strcat)
  For a safe version, see [strncpy](#strncpy)
  For a safe version that guarantees '\0' termination, see [FlyStrZCpy](FlyStr.html#FlyStrZCpy)
*/
char * strcpy(char *dst, const char *src)

Best Practices

Keep source code documentation minimal. Keep it to a brief explanation of the function or modules's purpose, parameters and return value(s) as a reference. You can always write a full tutorial book in some other tool if your APIs need it.

Only document public APIs. Private functions and methods classes and modules may have comments in the source code, but don't put them in flydoc special comments. Good coding practice hides implementation details.

flydoc special comments should be in in the source files, not header files, as flydoc doesn't search .h or .hpp files by default.

Stick to the basics for your markdown: text, bullets, ordered lists, tables, code blocks, images and links. The markdown engine is pretty good, but some corner cases may break it.

Only define one @defgroup or @class per module or class, as any others of the same name will be ignored. If a module or class is spread across multiple source files, use @inclass or @ingroup at the top the other source files.

Images

Markdown uses the following form for image links:

![Alt text](https://somelocation.com/folder/file.png "a title")

You can use a relative link if your image files are on the same site, for example in the same folder as your .html files:

![](file.png)

Since flydoc understands W3.CSS, it searches the [Alt text] for W#.CSS keywords. For example, if you want your logo to be converted to a circle and in grey scale, use:

![w3-circle w3-grayscale-max](mypic.jpg "my title")

The output HTML would be:

<img src="mypic.jpg" class="w3-circle" title="my title" alt="mypic.jpg">

See https://www.w3schools.com/w3css/w3css_images.asp

C Documentation Example

#include "maths.h"

/*!
  @mainpage Maths
  @version  0.42

  A multi-language math library that may someday rival MATLAB

  @defgroup maths A language independent API for higher mathematics

  Works across the top languages of today: C, C++, C#, Go, Java, Javascript, Python, Ruby, Rust,
  Swift and Typescript

  @example maths  An example of calculating area.

      #include "maths.h"
      int main(void) {
        printf("Area of 5x4 rectangle = %u\n", area(h,w));
        return 0;
      }
*/

/*!
  Calculates area of a rectangle
  @param  h   height
  @param  w   width
  @return area of rectangle
*/
unsigned area(unsigned h, unsigned w)
{
  return h * w;
}

/*!
  Calculates volume of a cube

  @example volume   An example of calculating volume.

      printf("volume of 3x4x5 cube = %u\n", volume(3,4,5));

  @param  h   height
  @param  w   width
  @param  d   depth
  @return volume of the cube
*/
unsigned volume(unsigned h, unsigned w, unsigned d)
{
  return h * w * d;
}

int main(void)
{
  printf("Area of 5x4 rectangle = %u\n", area(h,w));
  printf("volume of 3x4x5 cube = %u\n", volume(3,4,5));
  return 0;
}

C++ Documentation Example

#include <iostream>
using namespace std;

/*!
  @class Cars A class for listing automobiles

  Eventually target market will be car and dealerships and repair shops.
*/
class Car {
  public:
    string brand;
    string model;
    int year;

    /*!
      Constructor for Car
      @param    _brand    brand of car (e.g. Honda)
      @param    _model    model of car (e.g. Ridgeline)
      @param    _year     year of car (e.g. 2022)
    */
    Car(string _brand, string _model, int _year) {
      brand = _brand;
      model = _model;
      year = _year;
    }
    void Print(void);
};

int IsCarNewish(int year)
{
  return (year >= 1999);
}

/*!
  Print out make, model, year of the car
  @return none
*/
void Car::Print(void)
{
  cout << this->brand << " " << this->model << " " << this->year;
  if(IsCarNewish(this->year))
    cout << " new(ish)";
  cout << "\n";
}

int main() {
  // Create Car objects and call the constructor with different values
  Car carObj1("BMW", "X5", 1999);
  Car carObj2("Ford", "Mustang", 1969);

  // Print values
  cout << carObj1.brand << " " << carObj1.model << " " << carObj1.year << "\n";
  cout << carObj2.brand << " " << carObj2.model << " " << carObj2.year << "\n";
  carObj1.Print();

  return 0;
}

}
